XML (short for eXtensible Markup Language> is a technology standard for encoding structured data as simple text, in a way that is universal, transportable, and technology independent. XML files are readable and editable by any common text-editing application, and are usable by any text-aware application. The representation of structure is an important element of XML - it provides a mechanism to simply express many ways of organizing and structuring data, including expressions that meaningfully translate back-and-forth between XML and modern databases. Hauptwerk ODFs are XML documents that follow a set of standards defined by HW that closely correlate with database concepts such as Tables, Records, and Fields.

XML broadly places text into two categories: markup and data. Markup controls the identification of data, and gives it context and structure. Data is the raw informational content. Generally, in parts of an XML document that are not Data, extra white-space (blanks and tabs) and line-breaks placed between elements do not affect the interpretation. These extra characters may be used liberally to increase human readability. However, inside of Data, or within "quoted text" strings in Markup, white-space does matter, as it becomes a significant part of the Data or of the text-string.

In the definitions below, single-quotes (') identify elements used in XML, the quotes separating them from their descriptive text. However, these single-quotes do not appear in actual XML - only the text between the quotes. In contrast, double-quotes (") are commonly used to delimit string-literals in XML, and when seen below, would carry forward exactly as seen into the real XML text.
Tag:   
A markup string starting with '<', and ending with '>'. There are several types of Tags that are used in XML.
Example: '<ImaTag>'

Start-Tag:
The opening tag of an Element e.g., '<Pitch>'. Contains the Element's Name immediately following the opening '<'. Always paired with a a matching End-Tag.

End-Tag:
The closing tag of an Element e.g., '</Pitch>'. Contains the same Name as its paired Start-Tag, preceded by a forward-slash: '/'

Name/Type:
The first token between the markup-brackets of a Tag: in the prior two examples, 'Pitch'. Names are case-sensitive. The Name of an Element identifies its Type, which can be associated with rules for acceptable syntax and structure of the Content within an Element of that particular Type. Names do not contain embedded white-space, as that would immediately terminate the Name; however, the underscore character ('_') can be used within a name to emulate a blank for clarity. For example, 'TwoWords' and 'Two_Words' are both valid (but different!) Names, while using 'Two Words' where a Name is required would be interpreted as the Name being 'Two'.

Element:
A string consisting of a Start-Tag ('<xyz>'), and ending with a matching End-Tag ('</xyz>'). There may or may not be text between a Start and End-Tag pair.

Content:
The text found between an Element's Start and End-Tags; the Content may be null/empty. Content may include one or more nested Elements, in addition to Data text. Such nested Elements are called Child Elements, which are always wholly contained within their Parent Element. There may be one or more sibling Child Elements within their Parent's Content: if so, they appear one after the other, a successor Child's Start-Tag coming after its predecessor's End-Tag. White-space and line-breaks between siblings are acceptable but ignored, and may be freely introduced for clarity.

Example: '<o><a>123</a><b>ABC</b></o>' demonstrates a Parent (outer) Element 'o', whose Content consists of two Child Elements, 'a' and 'b'. The Content of Element 'a' is '123'; of 'b' is 'ABC'.

Elements may be nested to any depth. Nested Elements must be wholly contained within their Parent's Content, and sibling Elements may never overlap. The rules for correct nesting are the same as seen in most hierarchical lists, such as in word-processors (multi-level lists) and programming languages (nesting of blocks.)

Example: '<o><a>123</a></o>' is OK (wholly nested), but '<o><a>123</o></a>' is incorrect, as the Parent 'o' cannot end before its Child 'a' ends. Similarly, '<o><a>123</a><b>ABC</b></o>' is syntactically correct, as the siblings 'a' and 'b' do not overlap, whereas '<o><a>123<b></a>ABC</b></o>' is incorrect, as the siblings are overlapped.

Attribute:
Optional Name/Value pairs contained in a Start-Tag, following the Element's Name. Attributes are separated from the Element Name and from each other by white-space. Their syntax is 'attribute_name="attribute_value" '. A Start-Tag may contain 0, 1, or more Attributes.

Example: '<image source="picture.jpg" border="None">Stuff...</image>' is an Element named 'image' with two Attributes ('source' and 'border') whose values are respectively 'picture.jpg' and 'None'. The Content of 'image' is 'Stuff...'.

Analogs to Database Concepts:
The XML constructs used by HW in an ODF have very close correspondence to common database concepts. The outermost Elements define ODF Sections, directly analogous to database Tables. Almost every Section implements a "Record" structure, using 'o' Child Elements ('<o>...</o>') to correspond one-to-one with database Records inside a Section Table. Inside these 'o' Elements, HW uses other Tag-Names to define the analog of database "Fields" within the Parent Record Element. The various Field Elements within a single Row/Record are sibling Children of the Parent Record Element; and the sibling Row Elements within a Section Element are the Children comprising that Parent Section.
