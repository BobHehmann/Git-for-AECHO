AECHO V2 Help - Tracing a Sample

<Return to Help's Home Page>

The Sample Section enumerates all of the sound samples that the instrument has at its disposal. Each Record in this Section contains a minimal amount of data, usually three or four Fields: a SampleID, which is the unique Primary-key that identifies the sample throughout the rest of the ODF; the InstallationPackageID, which identifies of instrument that supplies the sound sample, which may be the ODF's own instrument, or another instrument altogether, allowing sounds to be borrowed; the directory path and filename of the sound sample, relative to the owning instrument's data directory; and less commonly, a tag that identifies License requirements for accessing the sample.

Before sounding, HW applies extensive processing to render the actual output waveform to be sent to the internal mixer: attack, decay, tremulants, wind-effects... The information that defines these processes is mostly stored in other ODF Sections, being linked to the samples by shared key values, much like how relationships are modeled in relational databases. Sample Tracing provides a tool to help understand some of these primary relationships, by tracing several of these linkages and presenting related Records. For example, starting with a sample from the Sample Section, Trace will locate linkages to and render related Records for:
  • Attack (Section Pipe_SoundEngine01_AttackSample)
  • Release (Section Pipe_SoundEngine01_ReleaseSample)
  • Tremulant processing for the Fundamental Pitch (Section TremulantWaveform)
  • Tremulant processing for the Third Harmonic (Section TremulantWaveform)
  • The sample's Source Instrument (RequiredInstallationPackage)

A second difficulty in understanding how sound will be processed based on casual inspection of an ODF is caused by the extensive reliance upon implicit defaults. Several of the Sections named above are comprised of a great many Fields, but often their ODF Records only carry Tags for a small number of Fields: the rest are filled in by implicit default values, which are often significant to the processing. Compressed-ODFs do not contain references to these implicit, default-valued Fields. Tracing fills in these missing details, by presenting every possible Field defined for a particular Section's Records, and showing both explicitly defined Fields whose values are taken directly from the ODF, and implicit default values, supplied by HW internal logic when a given Field does not appear in a Record, or appears but is without a value.

There are two ways to access Sample Tracing: through the menu item menu > Tools > Trace a Sample; through the Trace Sample button that is displayed whenever the selected Row/Record is a Sample Record. Both methods invoke the same Sample Trace form. They differ in that when the user invokes Tracing through the menu, the form's input field for Sample ID is initially blank, and the user is responsible for typing or pasting in the desired SampleID, then clicking the Trace form's Trace button to initiate the Trace; when using the main form's Trace Sample button, AECHO copies the SampleID from the current ODF record into the Sample Trace form's input field and initiates the Trace directly - saving keystrokes and reducing the chance entering the wrong SampleID.

This next picture starts with the sample sample, and shows the Sample Trace form that was created when the user clicked the main form's Trace Sample button:
Sample Trace Form


At the top of the form is the path & filename of the currently loaded ODF, and the Title "Sample Trace": these remain permanently displayed, even when the form is cleared. The Blue and Green lines are the Sections, starting with the Sample itself (Blue), followed by commonly linked Sections (Green). Each Section is listed, along with its starting and ending Line Numbers in the ODF. If there are no Records within a Section that are linked with the Sample being Traced, the first indented line under the Section's header states this, in Red: in the example, there are no related Attack Records. However, if there is a related Record in one of these Sections, then that Record's Fields are listed exhaustively. In this example, there is a related Release Record. The first indented line under Release tells us that the SampleID Field, defined by Tag "<c>", is present in the ODF, with a value of "400497". A few lines lower, it tells us that the Field LoadSampleRange_StartPositionValue, defined by Tag "<e>" is not present, so HW will use the default value of "0". Though scrolled off the example form's window, this sample went on to identify information regarding the two uses of Tremulants, and the source instrument package. Fields that explicitly appear in the ODF are displayed in Black, those that are not, and are therefore operating with default values, an in Brown.

At any time, the user can enter another SampleID into the textbox, and initiate another Trace. The results of each subsequent Trace are appended to the bottom of the display, with a blank line separating Traces. Clicking the menu item menu > Clear Traces will reset the display area to just its headers, showing the current ODF file that is loaded, and the Title "Sample Trace". The menu item menu > Print Traces... opens an operating-system standard print-dialog, allowing the user to print the entire contents of this forms content area, retaining all formatting. The menu item menu > Exit closes the Sample Trace form, clearing its contents.
